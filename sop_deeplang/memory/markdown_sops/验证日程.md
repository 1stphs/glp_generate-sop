第一部分：核心准则
1. 静态字段 vs 动态字段
   - 静态（直接从方案复制，不随实验进展改变）：{{Protocol_Approval_Date}}、{{Planned_Start_Date}}、{{Planned_End_Date}}、{{Planned_Report_Draft_Date}}、方案内的注释文本（如“时间表为预估…”）。
   - 动态（必须从原始记录/报告中提取）：{{Actual_Start_Date}}、{{Actual_End_Date}}、{{Report_Finalization_Date}}。
2. 必守规则
   - 报告中必须同时列出“方案值（Planned）”与“实际值（Actual）”，并明确标注各自来源（例如：Protocol – page X；Study raw records – 文件名/日期）。
   - 若方案中存在关于日程为预估的明确注释（文本完全可被检索到），报告在出现日程差异时应直接引用该注释原文并使用该注释作为是否需要对方案做变更说明的依据。
   - 对所有日期差异必须计算并报告差异天数（以整数天表示，正数表示实际晚于计划，负数表示实际早于计划）。
   - 若方案未包含“日程为预估”或类似注释，且出现重大日程偏差，报告应：列出事实、引用原始记录中任何关于变更的说明；若无说明，则将该项标记为需进一步追溯（示例：提交QA/Study Director 注释）。
3. 证据链与可追溯性
   - 每一项日期在报告中出现时，必须在同一段落或注释中标出其来源（协议编号/页面或实验日志文件名/记录页码/签名时间）。
4. 计算与保留规则
   - 日期差异以天为单位计算，使用公历日期差（见“计算逻辑”部分），结果保留整数天；不使用小时为单位，除非原始记录仅以小时计并且差异 <1 天需要说明，则在附录给出小时差并注明计算方法。

第二部分：通用模板（可由AI Generator直接执行的脚本指令）
步骤 A — 提取字段（数据源优先级）
1. 从Protocol文档提取并保存：
   - {{Protocol_Approval_Date}}
   - {{Planned_Start_Date}}
   - {{Planned_End_Date}}
   - {{Planned_Report_Draft_Date}}
   - {{Protocol_Schedule_Note}}（如存在）
2. 从Report文档或原始实验记录提取并保存：
   - {{Actual_Approval_Date_in_Report}}（若有）
   - {{Actual_Start_Date}}
   - {{Actual_End_Date}}
   - {{Report_Finalization_Date}}
   - 若存在实验日志或签名页，记录对应文件名/页码作为来源。

步骤 B — 校验与标记（静态/动态二元差异定位）
1. 对每一对应字段（Approval, Start, End, Report_Draft）判断是否相同：
   - 若 {{Protocol_Field}} == {{Report_Field}} → 标记为“匹配”（Match）。
   - 若不相同 → 标记为“差异”（Delta），并进入步骤 C 处理。
2. 若 Protocol 包含 {{Protocol_Schedule_Note}}（非空文本），将其全文摘录并作为可选注释文本（Quote_Source = "Protocol"）。

步骤 C — 计算逻辑（原子化）
1. 日期标准化：将所有日期转换为 ISO 格式 YYYY-MM-DD。
2. 计算差异天数：
   - Delta_Days = (Actual_Date as date) - (Planned_Date as date)
   - 表示方式：若 Delta_Days > 0 → “晚于计划 {{Delta_Days}} 天”；若 Delta_Days < 0 → “提前 {{abs(Delta_Days)}} 天”；若 0 → “与计划相同”。
   - 保留规则：以天为单位，结果为整数；若需小时级别，另写附录并说明数据来源与计算方法。
3. 记录计算时间戳与执行者（AI/自动化脚本）以满足可追溯性。

步骤 D — 生成报告文本（占位符化模板）
1. 基本陈述（用于所有字段；用中文模板示例）：
   - “方案批准日期（Protocol）: {{Protocol_Approval_Date}}（来源：{{Protocol_File}}）；报告中记录的批准日期：{{Actual_Approval_Date_in_Report}}（来源：{{Report_File}}）。[若匹配则写：两者一致。若不一致则写：差异为 {{Delta_Days_Approval}} 天，详情见证据链。]”
2. 对“实验开始/结束/报告完成”专段（以结束日期为例）：
   - “拟定实验结束日（Protocol）：{{Planned_End_Date}}（来源：{{Protocol_File}}）；实际实验结束日（Report/raw records）：{{Actual_End_Date}}（来源：{{Report_File_or_Log}}）；实际晚于/早于计划 {{Delta_Days}} 天。”
3. 若方案含有时间表为预估的注释（{{Protocol_Schedule_Note}} 非空）：
   - 追加段落（直接引用协议原文）：“方案注释：‘{{Protocol_Schedule_Note}}’。基于该注释，日程变动不必作为方案修订；本报告保留上述事实记录并提供原始证据链。”
4. 若方案无类似注释且出现差异：
   - 追加说明模版（若报告中含变更理由则摘录）：“报告中记录的变更理由：{{Reported_Change_Reason}}（来源：{{Report_File_or_Log}}）。若无变更理由，已将该项标记为需进一步追溯（见QA待办事项）。”
5. 引用格式与位置：
   - 每处日期后括注来源，例如：“（来源：Protocol – Section X；或 Raw Log – 文件名，页码）”。
6. 输出字段（结构化）：
   - JSON 或表格形式同时生成以便存档：
     {
       "Protocol_Approval_Date": "{{Protocol_Approval_Date}}",
       "Planned_End_Date": "{{Planned_End_Date}}",
       "Actual_End_Date": "{{Actual_End_Date}}",
       "Delta_Days_End": "{{Delta_Days}}",
       "Protocol_Schedule_Note": "{{Protocol_Schedule_Note}}",
       "Evidence_Refs": ["{{Protocol_File}}","{{Report_File}}","{{Log_File}}"]
     }

步骤 E — 分支与告警
1. 若任何关键字段（Approval、Start）在协议与报告中不匹配且没有书面解释，则生成“需追溯”告警并在报告附录记录：
   - “字段：{{Field_Name}}，协议值：{{Protocol_Value}}，报告值：{{Report_Value}}，动作：提交QA调查/联系Study Director。”
2. 若仅为计划性预估偏差且协议包含明确注释，则不自动要求方案修订，仅在报告中记录并引用注释。
