第一部分：核心准则
- 定义：该章节撰写时必须遵守的强制规则。
1. 二元差异定位：严格区分静态文本（字段标签、标题——直接复用方案中固定表述）与动态数据（姓名、电话、电子邮件等——必须从原始记录提取并可追溯）。
2. 可追溯性要求：每一项动态数据必须记录其来源文件名、页码/位置、提取时间与提取者（形成审计条目）。若有多处来源，则必须进行比对并记录比对结果与最终判定理由。
3. 优先级与仲裁：如方案(Protocol)与报告(Report)值不一致，优先采用已签署/批准的原始文件中的值；若无已签署版本，采用最近日期的经验证记录；仲裁过程必须形成可审计的登记条目。
4. 不可虚构：严禁填入未在原始材料中出现的联系人信息；若在所有原始材料中均缺失，则按照“强制空数据阻断”输出指定文本。
5. 格式一致性：姓名原文呈现（不做意图性简写）；电话保留原始呈现（不得随意更改数字）；电子邮件保留原文大小写（以原始记录为准）。
6. 计算逻辑：本章节不涉及数值计算；如未来需做格式转换或正则校验，须在SOP中逐步原子化说明保留位/校验规则并记录结果。

第二部分：通用模板
- 定义：可直接由 AI Generator 执行的脚本指令（使用占位符，禁止硬编码具体值）。
步骤：
1) 定位源区段
   - 从输入集合中查找标题包含“委托方负责人”或同义表述的区段：
     - {{Protocol_Source_File}} -> 定位到协议中文档/位置 {{Protocol_Location}}
     - {{Report_Source_File}} -> 定位到报告中文档/位置 {{Report_Location}}
2) 数据抽取（动态字段）
   - 提取字段：
     - 姓名 -> {{Client_Name}}
     - 电话 -> {{Client_Phone}}
     - 电子邮件 -> {{Client_Email}}
   - 对每个字段，记录审计条目模板：
     - Field: {{Field_Name}}
     - Protocol_Value: {{Protocol_Value_or_EMPTY}}
     - Report_Value: {{Report_Value_or_EMPTY}}
     - Final_Value: {{Final_Value_or_EMPTY}}
     - Source_File(s): {{Source_File_List}}
     - Source_Location(s): {{Source_Location_List}}
     - Extracted_By: {{Operator}}
     - Extracted_On: {{ISO8601_Timestamp}}
3) 值比对与仲裁逻辑
   - 若Protocol_Value == Report_Value且非空 -> Final_Value = Protocol_Value（记录一致）。
   - 若二者不一致：
     - 若存在已签署/批准的Protocol版本 -> Final_Value = Protocol_Value；记录仲裁理由。
     - 否则，选择最近的经验证来源（按文件日期）作为Final_Value；记录仲裁理由与来源。
   - 若所有来源均为空/不存在 -> 终止并输出单行：该章节在原始材料中无明确数据支持
4) 输出填充
   - 使用以下占位符模板生成最终报告区段（保持HTML或目标格式一致）：
     - <p>**姓名**：{{Client_Name_Final}}</p><p></p><p>**电话**：{{Client_Phone_Final}}</p><p></p><p>**电子邮件**：{{Client_Email_Final}}</p>
   - 同时输出对应的审计条目（结构化JSON或表格），字段对应上述审计条目模板。
5) 验证与记录
   - 对输出的每个字段执行空值检查与基本正则校验（电话字符为数字与允许的符号；电子邮件包含“@”）；将校验结果写入审计条目。
   - 将最终生成的区段与原始Protocol/Report的定位信息一并存档，供审计检索。
6) 失败分支
   - 若进入第3步判定所有来源均空 -> 仅输出：该章节在原始材料中无明确数据支持
   - 任何仲裁或人工干预均须在审计条目中写明操作者与时间。
