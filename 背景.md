 SOP 生成整个工作流的核心逻辑：

### 1. 为什么要生成 SOP？用来干嘛的？
**目的：**
在 GLP（良好实验室规范）或者医药临床前报告等严谨的场景中，报告的撰写往往具有极高的标准化要求（例如表格不能乱用格式、数字及浓度指标必须100%准确提取、不可随意换行和加标点等）。
**作用：**
SOP（Standard Operating Procedure，标准化操作规程）生成的目的，就是**对从以往大量人类撰写的优秀报告中进行“逆向工程”**。系统通过对比“报告原本的输入素材”和“最终人类写出的优质总结”，自动提炼出一条条“死规则”（即 SOP）。
以后再遇到类似的新鲜数据时，AI 就可以直接遵循这些被提炼出来的 SOP 规则来生成正文，从而保证输出结果的严谨性、格式的绝对一致性和内容的专业性。

---

### 2. 生成 SOP 需要什么条件和输入？
SOP 生成通常以**章节（Chapter）**为粒度来进行。主要的输入流来自 API 端点 (`POST /templates/sop_generation`)。

每一个需要生成 SOP 的章节都需要具备以下**最核心的输入条件**：
*   **`original_content`（方案原始内容）**：就是最初未经处理的原始数据材料（比如验证方案原文、Excel解析出来的数据等）。
*   **`target_generate_content`（目标报告内容）**：就是最终期望生成的、标准的、正确的报告结果（通常是来源于历史优质案例或者专家改好的正确范本）。
*   **`section_title`（当前处理章节的标题）**：为了让 AI 明确当前在处理报告的哪一部分。

---

### 3. SOP 生成的流程是怎样的？
整个生成流是一个经典的 **Map-Reduce（分散并发处理再汇总）** 加 **闭环自我修正（Actor-Critic）** 架构。

**全局流程：**
1.  **初始化与扇出 (Fan-out)**：系统拿到包含多个章节的列表后，为每一个章节分配一个独立的子工作流（多线程并发，大家互不干扰去生成自己的 SOP）。
2.  **子图处理 (闭环修正核心)**：
    *   **步骤 A - SOP Writer (生成者)**：AI 拿到原始内容和期望的目标内容，开始逆向思考：“我是按什么规则才把原始内容变成了目标内容？”，然后它会产出第一版带有核心规则和通用模板的 `current_sop`。
    *   **步骤 B - Simulator (模拟器_盲测)**：为了防止 Writer 胡说八道，系统找来一个不知情的模拟器。模拟器**只能看到原始内容和刚才那份生成的 SOP，绝对不给它看期望的目标内容**。模拟器根据这份 SOP 从原始内容里“做题”，产出一份“模拟生成内容”。
    *   **步骤 C - Reviewer (审查员)**：审查员同时拿到“盲测做出来的结果”和“期望的目标结果”进行严格的字符串级别的苛刻对比（查格式、查排版、查遗漏）。
        *   **如果两者一致 (Passed)**：说明这份 SOP 是真正可执行的，通关！保存输出。
        *   **如果不一致 (Failed)**：审查员会给出“报错原因 (feedback)”，然后把作业打回去给 **步骤 A（SOP Writer）** 重新修改 SOP 规则，再次进行盲测，最多允许重试 2 次（总尝试 3 次）。
3.  **如果重试 3 次依然失败**：系统会将其标记为 `Need Human Review`（需要人类介入）。
4.  **汇总聚合 (Reduce)**：所有并发出去的章节子工作流（无论成功还是超时失败）汇合在一起，执行 `aggregation`。

---

### 4. 最终输出什么？
最终整个工作流会输出一个大的 JSON 结构（在系统里会落盘到本地的 `tests/output/completed_chapters_{id}_{time}.json`，在 ACE 中则写入其存储系统或挂载点），里面是一个数组。
每一个数组元素就是一个章节的处理结果，包含：
*   章节 ID 及其基本信息。
*   **最终生成的 SOP (`current_sop`)**：无论过没过，里面都会有这份格式化好的三大段规则（涵盖：核心填写规则、通用模板、标准示例）。
*   **归纳的 SOP 类型 (`sop_type`)**：例如它是简单的文本插入（`simple_insert`）、规则模板（`rule_template`）还是复杂的复合逻辑（`complex_composite`）。
*   **最终状态 (`status`)**：是完美通过了盲测 (`PASSED`)，还是因为太难了卡住了需要人看 (`Need Human Review`)。

你可以把这套流程看作是“找一个老师（Writer）写本教材（SOP），找个学生（Simulator）闭卷考试，再找个教导主任（Reviewer）对答案。答案不对就让老师改教材，直到学生能闭卷考满分为止”。这就是保证严谨报告能够被自动化批量生成的底座。
